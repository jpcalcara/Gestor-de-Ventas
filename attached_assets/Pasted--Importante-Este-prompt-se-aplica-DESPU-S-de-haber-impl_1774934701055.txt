## Objetivo general
 
Agregar tres capacidades nuevas al sistema:
1. **Autoregistro público** — cualquier negocio puede registrarse solo sin intervención del superadmin
2. **Pagos con Mercado Pago** — suscripciones mensuales recurrentes por plan, con período de gracia ante falta de pago
3. **Sistema de planes y feature flags** — el superadmin define qué funcionalidades están disponibles en cada plan, y eso se aplica en tiempo real en frontend y backend

### 2.2 Middleware de feature flag para endpoints
 
Crear middleware factory en `server/features.ts`:
 
```typescript
export function requireFeatureMiddleware(featureKey: string) {
  return async (req: Request, res: Response, next: NextFunction) => {
    if (req.session.userRole === "sistemas") return next(); // sistemas siempre pasa
    
    const businessId = req.session.businessId;
    if (!businessId) return res.status(403).json({ message: "Sin negocio en sesión" });
 
    // Verificar también que el negocio esté activo (no suspendido)
    const business = await storage.getBusiness(businessId);
    if (business?.subscriptionStatus === "suspended") {
      return res.status(402).json({ 
        message: "Suscripción suspendida. Por favor renovar el plan.",
        code: "SUBSCRIPTION_SUSPENDED"
      });
    }
 
    const { features } = await getBusinessFeatures(businessId);
    if (!features[featureKey]) {
      return res.status(403).json({ 
        message: "Esta funcionalidad no está disponible en tu plan actual.",
        code: "FEATURE_NOT_IN_PLAN",
        feature: featureKey
      });
    }
    next();
  };
}
```

export function useFeatures() {
  const { data } = useQuery({
    queryKey: ["/api/my-features"],
    staleTime: 5 * 60 * 1000, // cache 5 minutos

### 5.4 Procesamiento de webhooks de Mercado Pago
 
En `server/mp-webhooks.ts`, crear función `processMPWebhook(event)` con esta lógica:
 
**Evento: pago aprobado (`payment` con status `approved`)**
- Buscar el negocio por `mpSubscriptionId`
- Setear `subscriptionStatus = "active"`
- Actualizar `lastPaymentAt = now`
- Calcular `nextPaymentAt = now + 1 mes`
- Limpiar `graceEndsAt`
- Registrar evento "payment_success" en `subscriptionEvents`
 
**Evento: pago rechazado o fallido (`payment` con status `rejected` o `cancelled`)**
- Buscar el negocio
- Si `subscriptionStatus` ya es "grace_period": no hacer nada (ya está en gracia)
- Si `subscriptionStatus` es "active": setear `subscriptionStatus = "grace_period"`, `graceEndsAt = now + gracePeriodDays días`
- Registrar evento "payment_failed" y "grace_period_started"
 
**Evento: suscripción cancelada (`subscription_preapproval` con status `cancelled`)**
- Setear `subscriptionStatus = "cancelled"`, `isActive = false`
- Registrar evento "subscription_cancelled"
 
**Evento: suscripción pausada (`subscription_preapproval` con status `paused`)**
- Setear `subscriptionStatus = "grace_period"`, `graceEndsAt = now + gracePeriodDays días`
 
### 5.5 Job de verificación diaria (período de gracia)
 
Crear `server/subscription-checker.ts` con una función que se ejecute al arrancar el servidor y luego cada 24 horas (usar `setInterval`):

export async function checkExpiredGracePeriods() {
  // Buscar todos los negocios con subscriptionStatus = "grace_period" y graceEndsAt < now
  const expired = await storage.getBusinessesWithExpiredGrace();
  
  for (const business of expired) {
    await storage.updateBusiness(business.id, {
      subscriptionStatus: "suspended",
      isActive: false,  // bloquea el acceso completo
    });
    await storage.createSubscriptionEvent({
      businessId: business.id,
      type: "suspended",
      description: `Cuenta suspendida por vencimiento del período de gracia`,
    });
    // Opcional: enviar email de notificación (si hay SMTP configurado)
  }
}
```

  if (req.session.userRole === "sistemas") return next();
  
  const businessId = req.session.businessId;
  if (!businessId) return next(); // sin negocio en sesión, dejar pasar (el otro middleware lo atrapa)
 
  const business = await storage.getBusiness(businessId);
  if (!business) return res.status(404).json({ message: "Negocio no encontrado" });
 
  if (business.subscriptionStatus === "suspended") {
    return res.status(402).json({
      message: "Tu suscripción está suspendida. Por favor renovar el plan.",
      code: "SUBSCRIPTION_SUSPENDED",
    });
  }
  // trial, active, grace_period: dejan pasar
  next();
}
```
 
Aplicar `requireActiveSubscription` globalmente en `server/routes.ts` después de `requireAuth`, ANTES de los endpoints de negocio (no aplicar a `/api/auth/*`, `/api/plans`, `/api/webhooks/*`, `/api/invitations/accept/*`).

## Restricciones y consideraciones importantes
 
1. **No modificar la lógica de roles existente**. Los roles `sistemas`, `admin`, `vendedor` no cambian. El sistema de features es una capa adicional encima.
 
2. **El rol `sistemas` nunca es bloqueado** por feature flags ni por estado de suscripción. Siempre tiene acceso total.
 
3. **Cache de features**: la consulta a `getBusinessFeatures` puede hacerse en cada request o cachearse con un TTL corto (5 minutos). Dado que Neon es serverless, preferir cachear en memoria con un Map simple `{ businessId -> { features, cachedAt } }` y refrescar si `cachedAt` tiene más de 5 minutos. Limpiar el cache del negocio cuando el superadmin cambia sus features.
 
4. **Webhook de MP es crítico**: siempre verificar la firma del webhook antes de procesar. MP envía un header `x-signature` que debe validarse con `MP_WEBHOOK_SECRET`. Si la firma no coincide, responder 401 y no procesar.
 
5. **El período de prueba (trial)** aplica a todos los registros nuevos, incluyendo plan Free. `trialEndsAt` se setea en el registro. Al vencer el trial sin suscripción activa, el negocio pasa a `grace_period` también (el job diario lo maneja).
 
6. **Ejecutar las migraciones** con `npm run db:push` después de todos los cambios de schema antes de arrancar.
 
7. **Mantener el diseño visual actual** de shadcn/ui, Inter font, TailwindCSS. Las páginas nuevas (register, pricing, billing, admin-plans) deben seguir el mismo sistema de diseño.
 
8. **La página `/register` y `/pricing` son públicas** — no deben tener el header/nav del sistema. Usar un layout alternativo sin navegación.

1. Schema completo (tablas `plans`, `features`, `plan_features`, `subscriptionEvents` + columnas en `businesses`) + seed inicial
2. Helper `getBusinessFeatures` y middleware `requireFeatureMiddleware`

9. Panel de billing `/billing` para el admin del negocio
10. Banner de estado de suscripción en el header

## Resultado esperado
 
Al terminar, el sistema debe permitir:
- Un negocio nuevo registrarse solo en `/register`, elegir plan, pagar con MP y empezar a usar el sistema sin intervención del superadmin
