### Estructura de roles actual
- `sistemas` → superadmin global (accede a todo)
- `admin` → administrador de sucursal/negocio
- `vendedor` → solo crea y edita ventas
 
### Lo que ya existe y funciona bien (NO tocar)
- Aislamiento de datos por sucursal (`branchId` como FK en productos, ventas, audit logs)
- Tabla `businesses` con campos: `id`, `razonSocial`, `cuit`, `encargado`, `telefono`, `mail`, `adminUserId`, `isActive`
- Tabla `branches` con FK a `businesses`
- Tabla `businessAdmins` (relación many-to-many usuarios-negocios)
- Tabla `userBranches` (relación many-to-many usuarios-sucursales)
- Middlewares `requireAuth`, `requireAdmin`, `requireSistemas` en `server/auth.ts`
- La sesión ya guarda: `userId`, `userEmail`, `userName`, `userRole`, `businessId`, `businessName`, `branchId`, `branchName`
 
---
 
## Objetivo
 
Quiero que el sistema funcione como un **SaaS multitenant real**, donde cada negocio (tenant) está completamente aislado de los demás. Un negocio no puede ver ni acceder a los datos de otro negocio bajo ninguna circunstancia. El usuario `sistemas` es el único con visibilidad global.

### CAMBIO 3: Sistema de invitaciones de usuarios por negocio
 
Actualmente solo el rol `sistemas` puede crear usuarios. Se necesita que el `admin` de cada negocio pueda invitar usuarios a su propio negocio mediante un sistema de invitaciones por email.
 
**En `shared/schema.ts`**, crear la nueva tabla:
 
```typescript
export const invitations = pgTable("invitations", {
  id: varchar("id").primaryKey().default(sql`gen_random_uuid()`),
  email: text("email").notNull(),
  businessId: varchar("business_id").notNull().references(() => businesses.id, { onDelete: "cascade" }),
  branchId: varchar("branch_id").references(() => branches.id, { onDelete: "set null" }),
  role: text("role").notNull().default("vendedor"), // "admin" | "vendedor"
  token: text("token").notNull().unique(),
  expiresAt: timestamp("expires_at").notNull(),
  usedAt: timestamp("used_at"),
  createdAt: timestamp("created_at").notNull().defaultNow(),
});
```
 
**Agregar los Zod schemas** correspondientes (`insertInvitationSchema`, etc.) en `shared/schema.ts`.
 
**En `server/storage.ts`**, agregar métodos:
- `createInvitation(data)`
- `getInvitationByToken(token: string)`
- `markInvitationUsed(id: string)`
- `getInvitationsByBusiness(businessId: string)`
 
**En `server/routes.ts`**, agregar los endpoints:
 
```
POST /api/invitations
  - requireAdmin
  - Solo puede invitar a su propio negocio (usar req.session.businessId)
  - Rol `sistemas` puede invitar a cualquier negocio
  - Genera token único (usar nanoid o crypto.randomUUID)
  - expiresAt = 72 horas desde ahora
  - Responde con la invitación creada (incluyendo el token para mostrar el link)
 
GET /api/invitations
  - requireAdmin
  - Devuelve todas las invitaciones del negocio en sesión
  - rol `sistemas` ve todas
 
DELETE /api/invitations/:id
  - requireAdmin
  - Solo puede borrar invitaciones de su propio negocio
 
GET /api/invitations/accept/:token  (endpoint público, sin requireAuth)
  - Busca la invitación por token
  - Verifica que no esté usada y no haya expirado
  - Devuelve los datos de la invitación (email, businessId, role) para pre-llenar el formulario de registro
 
POST /api/invitations/accept/:token  (endpoint público, sin requireAuth)
  - Recibe: firstName, lastName, password
  - Valida el token igual que arriba
  - Crea el usuario con los datos
  - Asigna el usuario al negocio via businessAdmins (si role = "admin") o userBranches (si role = "vendedor" y hay branchId)
  - Marca la invitación como usada (usedAt = now)
  - Inicia sesión automáticamente
```
 
**En el frontend**, agregar:
- Página pública `/invite/:token` que muestre el formulario de registro pre-llenado con el email de la invitación
- En la página de Usuarios (`client/src/pages/users.tsx`), agregar botón "Invitar usuario" visible para `admin` que abra un dialog con: campo email, selector de rol (`admin`/`vendedor`), selector de sucursal (si es vendedor). Al enviar, llama a `POST /api/invitations` y muestra el link generado para copiar.

### CAMBIO 4: Aislamiento total por `businessId` en todos los endpoints
 
Revisar **todos** los endpoints en `server/routes.ts` y asegurarse de que ningún `admin` o `vendedor` pueda acceder a datos de otro negocio. Las reglas son:
 
**Para rol `sistemas`**: puede ver y operar sobre todo sin filtro por negocio.
 
**Para rol `admin` y `vendedor`**: cada consulta debe filtrar por `req.session.businessId`. Si `req.session.businessId` está vacío para estos roles, devolver 403.
 
Los endpoints a revisar y corregir específicamente:
 
- `GET /api/businesses` → si es `admin`, devolver solo el negocio propio. Si es `sistemas`, devolver todos.
- `GET /api/branches` → si es `admin` o `vendedor`, filtrar por `req.session.businessId`
- `GET /api/users` → si es `admin`, devolver solo usuarios que pertenezcan a su mismo negocio (via `businessAdmins` o `userBranches` en alguna sucursal del negocio)
- `GET /api/audit-logs` → ya filtra por sucursal, pero agregar validación que la sucursal pertenezca al negocio en sesión
- `GET /api/products` → ya filtra por `branchId`, verificar que el `branchId` pertenece al negocio en sesión antes de devolver datos
 
**Agregar un helper en `server/auth.ts`**:
 
```typescript
// Verifica que un branchId pertenezca al negocio en sesión
export async function validateBranchBelongsToBusiness(
  branchId: string,
  businessId: string
): Promise<boolean> {
  const branch = await storage.getBranch(branchId);
  return branch?.businessId === businessId;
}
```
 
Usar este helper en cualquier endpoint donde se reciba un `branchId` por parámetro o body.

## Restricciones importantes
 
1. **No romper el aislamiento por sucursal que ya existe**. El `branchId` en sesión sigue siendo necesario y funcional. El `businessId` es una capa adicional encima.
 
2. **No modificar el flujo de selección de sucursal** que ya funciona en el frontend (el selector de sucursal al iniciar sesión). Solo asegurarse de que al seleccionar sucursal se valide que pertenece al negocio en sesión.
 
3. **No cambiar el sistema de roles**. Los roles `sistemas`, `admin`, `vendedor` se mantienen exactamente igual.
 
4. **Mantener el diseño visual actual** (Inter font, shadcn/ui, TailwindCSS, colores y espaciados existentes). Solo agregar componentes nuevos donde sea necesario siguiendo el mismo estilo.
 
5. **Ejecutar todas las migraciones de Drizzle** después de cada cambio al schema. Usar `npm run db:push` o el comando de migración configurado en el proyecto.
 
6. **Backward compatibility**: Los negocios y sucursales que ya existen en la base de datos no deben romperse. Para los registros existentes de `companySettings` sin `businessId`, asignarlos al primer negocio activo encontrado o dejarlos como null (el código los ignorará).
